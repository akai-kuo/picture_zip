## 逐段講稿重點

### 1. 開場（0:40）
- 這是什麼：一個圖片壓縮與轉檔工具
- 解決什麼痛點：上傳圖片太大、格式不對，想快速壓縮/轉檔又不想用來路不明的線上工具
- 團隊分工一句話帶過（前端 / 後端 / 部署）

### 2. 主流程操作（2:20）
邊做邊講，不要只是操作不說話：
1. 拖曳或點擊上傳一張圖 → 講「支援 JPG/PNG/WebP，單檔限制 5MB」
2. 調整壓縮品質滑桿 + 選輸出格式 → 講「這些設定會直接影響輸出檔案大小」
3. 按下「開始壓縮」→ 講「後端用 Sharp 處理」
4. 秀出結果：原始大小 → 壓縮後大小 → 節省百分比 → 下載

**建議**：這張示範圖片事先準備好，選一張壓縮效果明顯的（例如一張 3-5MB 的照片壓到 500KB 以內），視覺效果會比較有說服力。

### 3. 用心點 / 加分亮點（2:00）— 已定案

**主打：批次上傳體驗**
講稿方向：「大部分同類工具一次只能處理一張圖，但使用者常見的情境是一次有一整個資料夾的圖要壓縮。我們讓使用者一次最多選 20 張，每張圖獨立顯示狀態，就算某幾張失敗也不影響其他張，最後統計整批省下多少空間。」
操作：一次選 3-5 張圖（其中故意混一張超過 5MB 或錯誤格式的），展示各自獨立的狀態與統計摘要。

**次要亮點：貼上上傳（收尾用的小驚喜）**
講稿方向：「截圖後不用先存檔，直接 Ctrl+V 貼上就能上傳。」
操作：截一張圖，現場按 Ctrl+V，圖片直接出現在列表。

**補充（不用單獨佔時間，口頭帶過即可）**：錯誤訊息都轉換成看得懂的中文，不是技術性錯誤代碼。

### 4. 技術架構說明（1:30）
由負責後端/部署的組員講：
- 後端：Node.js + Express + Sharp 處理圖片
- 資料流：formidable 接收上傳 → Sharp 處理 → 回傳結果 → 前端顯示
- 部署：Docker 打包、GitHub Actions CI/CD、推送到 Docker Hub

> 如果後端到時候還沒做完，這段改成「規劃中的架構」講法：「我們設計的資料流是這樣，目前後端還在實作，前台已經先把介面跟串接邏輯準備好了」——誠實講完成度，比硬撐著假裝做完更有說服力。

### 5. AI 協作說明（1:00）
- 用了什麼 AI 服務、加速了哪些環節（架構規劃、前端頁面設計、錯誤處理邏輯）
- 自己判斷、跟 AI 建議不同的地方（例如選 formidable 而非 multer 的原因）
- 印象最深的一次對話（一句話）

### 6. 收尾 + QA（0:30）
一句話總結：「這是一個讓使用者不用註冊、不用等待，就能快速壓縮轉檔圖片的小工具」，然後開放提問。

## 備案模式：如果展示當天後端還沒完成

**方案：前端改用假資料模擬 API 回應**，讓主流程 Demo 照樣能跑完整流程，不會卡在「轉圈圈轉不出結果」。

作法：在 `compressImage()` 裡暫時加一個開關，展示時不打真的 API，改成 `setTimeout` 模擬處理時間後回傳假的 `{ status: 'success', data: {...} }`，數字可以用真實壓縮過的檔案大小去估算，讓畫面看起來合理。

這件事不用等後端，你今天就可以先做起來當保險。要我現在幫你把這個「模擬模式」寫進 `index.html` 嗎？
